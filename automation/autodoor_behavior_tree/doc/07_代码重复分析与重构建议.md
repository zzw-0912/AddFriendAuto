| `key_down` | 完整实现 | 完整实现 | 完整实现 | 100% 逻辑重复 |

**代码对比示例** (`key_press` 方法):

```python
# input_controller.py (标准版)
def key_press(self, key: str, action: str = "press", duration: int = 0) -> None:
    self._set_simulating(True)
    try:
        if action == "press":
            if duration > 0:
                pyautogui.keyDown(key)
                pyautogui.sleep(duration / 1000)
                pyautogui.keyUp(key)
            else:
                pyautogui.press(key)
        elif action == "down":
            pyautogui.keyDown(key)
        elif action == "up":
            pyautogui.keyUp(key)
    finally:
        self._set_simulating(False)

# input_controller_factory.py (工厂版)
def key_press(self, key: str, action: str = "press", duration: int = 0) -> None:
    import pyautogui
    self._set_simulating(True)
    try:
        if action == "press":
            if duration > 0:
                pyautogui.keyDown(key)
                time.sleep(duration / 1000.0)
                pyautogui.keyUp(key)
            else:
                pyautogui.press(key)
        elif action == "down":
            pyautogui.keyDown(key)
        elif action == "up":
            pyautogui.keyUp(key)
    finally:
        self._set_simulating(False)

# dd_input.py (DD版)
def key_press(self, key: str, action: str = "press", duration: int = 0) -> None:
    if action == "press":
        self.key_down(key)
        if duration > 0:
            time.sleep(duration / 1000.0)
        self.key_up(key)
    elif action == "down":
        self.key_down(key)
    elif action == "up":
        self.key_up(key)
```

**差异分析**:
1. **InputController**: 使用 pyautogui，有完整的模拟状态管理
2. **PyAutoGUIInput**: 与 InputController 几乎完全相同，只是导入方式不同
3. **DDVirtualInput**: 使用 DD64.dll，但接口逻辑完全一致

**影响**:
- 维护困难：修改一个控制器需要同步修改多个文件
- 代码膨胀：约 300+ 行重复代码
- 行为不一致风险：不同控制器可能有细微差异

**重构建议**:
1. 统一使用 `input_controller_factory.py` 作为唯一入口
2. 删除 `input_controller.py` 或将其实现为工厂模式的包装器
3. 提取公共逻辑到 `BaseInputController` 的模板方法

```python
# window_manager.py
@staticmethod
def get_window_rect(hwnd: int) -> Optional[Tuple[int, int, int, int]]:
    rect = wintypes.RECT()
    if user32.GetWindowRect(hwnd, ctypes.byref(rect)):
        return (rect.left, rect.top, rect.right, rect.bottom)
    return None

# window_capture.py
@staticmethod
def get_window_rect(hwnd: HWND) -> Optional[Tuple[int, int, int, int]]:
    rect = RECT()
    if user32.GetWindowRect(hwnd, ctypes.byref(rect)):
        return (rect.left, rect.top, rect.right, rect.bottom)
    return None

# coordinate.py
@staticmethod
def get_window_rect(hwnd: wintypes.HWND) -> Optional[Tuple[int, int, int, int]]:
```

**差异分析**:
- 三处实现几乎完全相同，仅类型提示略有差异
- WindowCapture 使用自定义 RECT 类，其他使用 wintypes.RECT

**影响**:
- 约 50+ 行重复代码
- 窗口相关 API 变更时需要多处修改

**重构建议**:
1. 统一使用 `WindowManager` 作为窗口操作的唯一入口
2. `WindowCapture` 和 `CoordinateConverter` 委托给 `WindowManager`
3. 或提取公共 Windows API 包装到单独模块

### 2.3 坐标转换重复

#### 2.3.1 问题描述

**涉及文件**:
- `bt_utils/coordinate.py` - CoordinateConverter 类
- `bt_utils/window_capture.py` - WindowCapture 类
- `bt_core/context.py` - ExecutionContext 类

**重复方法**:

| 方法 | CoordinateConverter | WindowCapture | ExecutionContext |
|------|---------------------|---------------|------------------|
| `screen_region_to_window` | 有 | 无 | 有（内联） |
| `window_region_to_screen` | 有 | 无 | 有（内联） |

| `client_to_absolute` | 有 | 无 | 无 |

**代码对比** (`convert_to_screen_coords` in context.py vs CoordinateConverter):

```python
# context.py
@staticmethod
def convert_to_screen_coords(region: tuple) -> tuple:
    if self._bound_window is None:
        return region
    from bt_utils.coordinate import CoordinateConverter
    if len(region) == 2:
        result = CoordinateConverter.client_to_absolute(region[0], region[1], self._bound_window)
        return result if result else region
    return CoordinateConverter.window_region_to_screen(region, self._bound_window)

# coordinate.py
@staticmethod
def window_region_to_screen(window_region: tuple, hwnd: int) -> tuple:
    left_top = CoordinateConverter.client_to_absolute(window_region[0], window_region[1], hwnd)
    right_bottom = CoordinateConverter.client_to_absolute(window_region[2], window_region[3], hwnd)
    if left_top and right_bottom:
        return (left_top[0], left_top[1], right_bottom[0], right_bottom[1])
    return (0, 0, 0, 0)
```

**影响**:
- 坐标转换逻辑分散在多个地方
- 窗口绑定功能的坐标转换路径不清晰

**重构建议**:
1. 统一使用 `CoordinateConverter` 作为坐标转换的唯一入口
2. `ExecutionContext` 完全委托给 `CoordinateConverter`
3. 添加坐标转换的单元测试确保一致性

## 4. 具体重构方案

### 4.1 输入控制器统一方案

**目标**: 合并 `input_controller.py` 和 `input_controller_factory.py`

**步骤**:
1. 保留 `input_controller_factory.py` 作为唯一实现
2. 将 `input_controller.py` 改为兼容性包装器
3. 更新所有导入引用

**代码示例**:

```python
# input_controller.py - 重构后
from bt_utils.input_controller_factory import InputController as _InputController

class InputController(_InputController):
    """兼容性包装器，委托给工厂实现"""
    pass
```

### 4.2 窗口管理统一方案

**目标**: 统一使用 `WindowManager`

**步骤**:
1. `WindowCapture` 委托窗口查询给 `WindowManager`
2. `CoordinateConverter` 委托窗口查询给 `WindowManager`
3. 提取公共 Windows API 常量

**代码示例**:

```python
# window_capture.py - 重构后
@staticmethod
def get_window_rect(hwnd: HWND) -> Optional[Tuple[int, int, int, int]]:
    from bt_utils.window_manager import WindowManager
    return WindowManager.get_window_rect(hwnd)
```

### 4.3 条件节点模板方法方案

**目标**: 提取公共逻辑到基类

**步骤**:
1. 在 `ConditionNode` 中添加 `validate_region` 方法
2. 添加 `handle_check_exception` 装饰器
3. 子类只实现核心检测逻辑

**代码示例**:

```python
# nodes.py - ConditionNode 重构后
class ConditionNode(Node):
    def _pre_check(self, context) -> bool:
        """前置检查，子类可覆盖"""
        if not self._validate_region():
            return False
        return True
    
    def _post_check(self, context, result: bool) -> bool:
        """后置处理，子类可覆盖"""
        return result
    
    @exception_handler("条件检测异常")
    def tick(self, context):
        if not self._pre_check(context):
            return NodeStatus.FAILURE
        result = self._check_condition(context)
        return self._post_check(context, result)
```

## 6. 总结

本项目存在多处代码重复，主要集中在：

1. **输入控制**: 3个类实现相同接口，约300行重复
2. **窗口管理**: 3个类实现相同功能，约100行重复
3. **条件节点**: 6个子类重复基类逻辑，约200行重复
4. **其他**: 单例、坐标、图像等约200行重复

**总计可优化代码**: 约800行

**建议优先级**:
1. P0: 输入控制器统一（影响最大）
2. P1: 条件节点模板方法（维护性提升明显）
3. P2: 其他低优先级项

通过系统性的重构，可以显著提升代码质量和维护效率。
