## 2. 项目目标

FriendAuto 是一个可交付客户使用的 Windows 桌面应用。用户安装后完成账号登录、设备绑定、试用或会员充值，再在桌面端配置自动加好友任务。桌面端启动本地 Python worker，worker 调用 AutoDoor 行为树项目执行自动化，结果返回桌面端并上报服务端。

项目边界：

- 本项目实现桌面端、后端、数据库、管理后台、设备绑定、会员/试用控制、订单、任务调度、运行记录和反馈。
- AutoDoor 自动化能力是外部项目，FriendAuto 只做桥接，不重写 AutoDoor 行为树引擎。
- 所有商业规则由服务端裁决，客户端只展示结果和触发操作。

## 4. 当前支付规划

当前正式流程是人工充值：

- 桌面端创建 `manual_wechat` 订单。
- 用户联系工作人员并完成线下/人工收款。
- 管理后台确认收款。
- 后端开通或延长会员。

后续大概率接入：

- 微信支付：商户配置、签名、回调验签、幂等、订单同步、审计。
- 支付宝支付：应用配置、签名、回调验签、幂等、订单同步、审计。

当前 mock callback 仅用于 debug，不代表官方支付已完成。

## 5. 关键业务规则

- 软件只处理客户自有或已授权的联系人数据。
- 一台设备可绑定多个账号；一个账号只能绑定一台设备。
- 新用户默认 20 次成功加好友试用额度。
- 只有 `event = success` 扣试用次数。
- 失败、无效、异常不扣试用。
- 会员有效时不扣试用。
- 会员续费从当前有效期末尾叠加。
- 套餐决定任务卡数量：无会员/月卡 1 张，季卡 2 张，年卡 3 张。
- 客户端不保存支付密钥，不裁决会员和试用。

## 6. 上线前路线

高优先级：

- 继续真实 AutoDoor 联调，验证窗口绑定、DPI、点击位置和结果统计。
- 补齐 PostgreSQL/RDS 迁移路径和 Alembic 迁移。
- 梳理官方微信/支付宝支付接入方案。
- Windows `.exe` 打包和干净 Windows 安装验证。

中优先级：

- 客户端自动更新。
- HTTPS 服务部署。
- 接口限流。
- 客户端日志导出。
- 异常告警和服务器监控。
- 用户协议、隐私政策、数据删除机制。
- 代码签名。

低优先级：

- 清理 legacy `SettingsPage.tsx`。
- 将多账号 token 存储迁移到 Tauri 原生存储。
- 为 worker 行为树补丁逻辑增加快照或单元测试。
