# FriendAuto 项目规划与交接文档

更新时间：2026-06-24  
项目目录：`D:\FriendAuto`

## 1. 项目目标

FriendAuto 是一款可上线交付客户使用的 Windows 桌面应用。客户通过 `.exe` 安装/打开软件后，完成邮箱注册登录、设备绑定、会员充值或试用，然后在桌面端配置每日自动加好友任务。

本项目只实现流程图中的黑色部分：桌面端、后端服务、数据库、支付、后台管理、设备绑定、会员/试用控制、脚本调度和运行记录。

流程图中的红色部分为已有自动化程序，不在本项目中重写。桌面端只负责向自动化程序传参、启动程序、接收运行结果、展示状态、上报服务器。

## 2. 阶段拆分

### 阶段 0：项目初始化与技术底座

目标：搭建能运行、能打包、能联调的基础工程。

主要任务：

- 创建桌面端工程：Tauri + React + TypeScript。
- 创建后端工程：FastAPI + PostgreSQL。
- 创建后台管理端，可与桌面端共用 React 技术栈，也可独立管理后台。
- 规划数据库迁移工具和环境配置。
- 配置本地开发环境、启动脚本、基础 README。
- 准备测试用 Python 自动化脚本：打开记事本并输入 `123456`。

验收标准：

- 桌面端可以本地启动。
- 后端可以本地启动并连接数据库。
- 桌面端可以调用后端健康检查接口。
- 桌面端可以启动测试 Python 脚本并收到成功/失败结果。

### 阶段 1：账号、登录与设备绑定

目标：完成客户打开软件后的基础准入流程。

主要任务：

- 邮箱验证码注册/登录。
- 登录态保存和 token 刷新。
- 生成本机机器码。
- 首次登录自动绑定设备。
- 一个账号默认只允许绑定一台设备。
- 换设备登录时提示联系管理员修改绑定。
- 后台支持查看、解绑、修改设备绑定。

验收标准：

- 未登录用户不能进入主界面。
- 首次登录可以绑定当前设备。
- 第二台设备登录会被拦截。
- 后台改绑后，新设备可以正常登录。

### 阶段 2：会员、试用与充值

目标：实现商业化控制逻辑。

主要任务：

- 进入主界面后自动弹出充值窗口。
- 未充值用户可以关闭充值窗口并试用。
- 默认试用额度为 20 次成功加好友。
- 每成功加 1 个好友扣 1 次试用。
- 失败、无效、已添加、搜索不到、脚本异常不扣试用次数。
- 会员有效时不扣试用次数。
- 会员过期后重新校验，必要时弹充值窗口。
- 套餐分为月卡、季卡、年卡三档，价格由后台配置。
- 接入微信官方支付和支付宝官方支付。
- 支付成功后由服务端开通或延长会员。

验收标准：

- 未充值用户进入后弹充值窗口。
- 关闭充值窗口后仍可试用。
- 成功扣次准确，失败不扣次。
- 试用 20 次成功额度用完后必须充值。
- 会员购买成功后立即恢复正常使用。
- 会员过期后再次使用会弹充值窗口。

### 阶段 3：主界面与自动化脚本联调

目标：完成用户真正操作任务的桌面端界面。

主要任务：

- 主界面支持输入当天要加多少人。
- 支持选择是否创建标签。
- 支持输入打招呼语或用户自定义文本。
- 支持开始、停止、查看运行日志。
- 支持展示成功数、失败数、无效数、剩余试用次数、会员到期时间。
- 桌面端传参给 Python 自动化程序。
- 自动化程序回传任务进度和结果。
- 结果上报服务端，服务端记录任务和明细。
- 用测试脚本完成联调后，再替换真实自动化脚本。

验收标准：

- 桌面端能启动测试脚本。
- 测试脚本能打开记事本并输入 `123456`。
- 桌面端能接收到脚本返回状态。
- 真实脚本替换后不需要改会员和支付逻辑。

### 阶段 4：后台管理与运营功能

目标：让项目具备上线后的售后、运营和管理能力。

主要任务：

- 用户管理：邮箱、注册时间、状态、最近登录。
- 设备管理：机器码、绑定时间、解绑、改绑、备注。
- 会员管理：开通、延期、冻结、恢复、查看到期时间。
- 套餐管理：月卡、季卡、年卡价格配置。
- 订单管理：订单号、用户、套餐、金额、支付方式、状态。
- 联系人数据库：微信昵称、微信号、标签、备注、状态。
- 任务日志：任务 ID、用户、设备、开始/结束时间、成功/失败/无效统计。
- 扣次记录：每次试用扣次都有明细。
- 管理员操作审计：记录谁在什么时候修改了用户、设备、会员或订单。

验收标准：

- 后台能查到用户、设备、订单、会员和任务。
- 后台能给用户改绑设备。
- 后台能配置三档套餐价格。
- 后台能人工延长会员或备注订单。

### 阶段 5：上线打包、安全与稳定性

目标：让软件可以真正交付客户使用。

主要任务：

- Windows `.exe` 打包。
- 客户端自动更新能力。
- HTTPS 服务部署。
- 支付回调验签和幂等处理。
- 客户端日志导出。
- 服务器异常、网络断开、脚本崩溃时的错误提示。
- 接口限流，防止刷验证码、刷订单、刷任务。
- 客户端代码签名，减少 Windows 安全提示。
- 用户协议、隐私政策、授权名单承诺。

验收标准：

- 干净 Windows 电脑可以安装并运行。
- 支付成功后会员状态正确。
- 网络异常不会绕过会员校验。
- 客户端崩溃或退出后不会重复扣试用次数。
- 后台能协助售后定位问题。

## 3. 项目关键决策

### 产品与业务规则

- 软件定位为处理客户自有或已授权的微信联系人名单。
- 不实现绕过微信验证码、限制、封控或风控的能力。
- 客户端每次开始任务前必须向服务器校验会员、试用、设备和版本状态。
- 未充值用户可试用 20 次成功加好友。
- 试用次数按“成功加好友”扣减，不按启动次数扣减。
- 会员有效时不扣试用次数。
- 会员过期后重新走试用/充值判断。
- 套餐为月卡、季卡、年卡三档。
- 一个账号默认绑定一台设备。

### 技术决策

- 桌面端：Tauri + React + TypeScript。
- 后端：FastAPI + PostgreSQL。
- 登录方式：邮箱 6 位验证码，无密码登录。
- 支付方式：微信官方支付 + 支付宝官方支付。
- 自动化程序：外部 Python worker，通过标准输入输出或本地 IPC 与桌面端通信。
- 首个联调脚本：打开记事本并输入 `123456`。
- 服务器时间为准，不信任客户本机时间。
- 会员、订单、试用次数、设备绑定均由服务端保存和裁决。

### 安全决策

- 客户端不保存真实会员判断结果，只展示服务端返回状态。
- 客户端不写死套餐价格。
- 支付密钥只放服务端。
- 支付成功以服务端支付回调验签为准，不能只信客户端轮询。
- 试用次数存在服务端，卸载重装不能重置。
- 机器码只上传哈希后的标识，不上传明文硬件信息。
- 关键写入接口必须做幂等，避免重复扣次或重复开通会员。

## 7. 架构思路

### 总体架构

```text
Windows 桌面端
  |
  | HTTPS API
  v
FastAPI 后端服务
  |
  v
PostgreSQL 数据库

Windows 桌面端
  |
  | 本地进程 / IPC
  v
Python 自动化程序
```

### 桌面端职责

- 提供用户界面。
- 完成邮箱验证码登录。
- 采集硬件信息并生成机器码。
- 展示会员、试用、设备状态。
- 展示充值弹窗。
- 调用服务端创建订单和查询订单。
- 启动本地 Python 自动化程序。
- 接收脚本运行结果并展示日志。
- 将任务结果上报服务端。

桌面端不负责最终判断会员是否有效，不保存真实试用次数，不保存支付密钥。

### 后端职责

- 用户注册登录。
- 邮箱验证码发送和校验。
- token 签发和刷新。
- 设备绑定和改绑。
- 会员状态判断。
- 试用次数扣减。
- 套餐价格配置。
- 订单创建和查询。
- 微信/支付宝支付回调验签。
- 联系人数据筛选。
- 任务日志保存。
- 后台管理接口。

后端是所有商业规则的唯一可信来源。

### 自动化程序职责

- 接收桌面端传入的任务参数。
- 执行本地自动化流程。
- 返回每个目标的执行结果。
- 返回成功、失败、无效、异常、完成等状态。

自动化程序不负责会员判断、充值判断、试用次数判断。

### 后台管理职责

- 管理用户。
- 管理设备绑定。
- 管理会员和套餐。
- 查看订单和支付状态。
- 查看任务日志和扣次明细。
- 修改用户备注。
- 处理人工补单、改绑、延期等售后操作。

## 9. 核心接口规划

### 账号与设备

- `POST /auth/send-code`：发送邮箱验证码。
- `POST /auth/login`：验证码登录或注册。
- `POST /auth/refresh`：刷新登录 token。
- `POST /devices/bind`：绑定当前设备。
- `GET /devices/current`：查看当前设备绑定状态。

### 会员与支付

- `GET /me/status`：获取会员、试用、设备、版本状态。
- `GET /plans`：获取可购买套餐。
- `POST /orders`：创建充值订单。
- `GET /orders/{order_id}`：查询订单状态。
- `POST /payments/wechat/callback`：微信支付回调。
- `POST /payments/alipay/callback`：支付宝支付回调。

### 任务与联系人

- `POST /tasks/start-check`：开始任务前校验。
- `GET /contacts/search`：按微信昵称和微信号筛选联系人。
- `POST /tasks/{task_id}/results`：上报单条执行结果。
- `POST /tasks/{task_id}/finish`：结束任务。

### 后台管理

- `GET /admin/users`：用户列表。
- `GET /admin/users/{user_id}`：用户详情。
- `PATCH /admin/users/{user_id}/membership`：修改会员。
- `GET /admin/devices`：设备列表。
- `PATCH /admin/devices/{device_id}`：修改设备绑定。
- `GET /admin/orders`：订单列表。
- `GET /admin/tasks`：任务列表。
- `GET /admin/audit-logs`：后台操作日志。

## 10. 脚本通信协议

### 桌面端传给脚本

```json
{
  "run_id": "task_001",
  "daily_limit": 20,
  "create_tag": true,
  "greeting_text": "你好，我是XXX",
  "contacts": [
    {
      "contact_id": "contact_001",
      "wechat_nickname": "张三",
      "wechat_id": "wxid_001"
    }
  ]
}
```

### 脚本返回给桌面端

```json
{
  "run_id": "task_001",
  "contact_id": "contact_001",
  "event": "success",
  "message": "添加成功",
  "timestamp": "2026-06-24T12:00:00+08:00"
}
```

支持事件：

- `started`
- `progress`
- `success`
- `invalid`
- `failed`
- `finished`
- `error`

扣次规则：

- 只有 `event = success` 时才扣试用次数。
- 每个 `run_id + contact_id` 只能扣一次。
- 重复上报必须幂等处理。

## 11. 风险与注意事项

- 不能把会员和试用判断放在客户端，否则容易被破解。
- 不能把支付密钥放在客户端。
- 不能只依赖客户端判断支付成功。
- 不能让卸载重装重置试用次数。
- 不能因为脚本重复上报而重复扣次。
- 不能在网络断开时允许启动新任务。
- 不能混淆失败、无效和成功，否则扣次会引发售后纠纷。
- 需要明确软件只处理客户自有或已授权数据。
- 需要准备用户协议、隐私政策、数据删除机制。
